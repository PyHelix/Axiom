## Step 3: Lint Check (CuPy Bug Detection)

Before dry-run, grep each script for known CuPy bugs that will crash on volunteer GPUs:

```bash
SCRIPT="/opt/axiom_boinc/html/user/experiments/your_experiment.py"
FAIL=0
if grep -Pn 'rng\.random\(' "$SCRIPT" | grep -v 'random_sample'; then
  echo "LINT FAIL: rng.random() does not exist in CuPy. Use rng.random_sample()"
  FAIL=1
fi
if grep -Pn 'xp\.linalg\.eig\b|cp\.linalg\.eig\b|cupy\.linalg\.eig\b' "$SCRIPT" | grep -v 'eigh'; then
  echo "LINT FAIL: cupy.linalg.eig() does not exist. Use numpy.linalg.eig(cp.asnumpy(m))"
  FAIL=1
fi
if grep -Pn 'xp\.linalg\.eigvals\b|cp\.linalg\.eigvals\b' "$SCRIPT" | grep -v 'eigvalsh'; then
  echo "LINT FAIL: cupy.linalg.eigvals() does not exist. Use numpy.linalg.eigvals(cp.asnumpy(m))"
  FAIL=1
fi
if grep -Pn 'cupyx\.scipy\.linalg' "$SCRIPT"; then
  echo "LINT FAIL: cupyx.scipy.linalg does not reliably exist. Use cupy.linalg or numpy.linalg fallback."
  FAIL=1
fi
if grep -Pn 'INITIAL_N.*=.*max\(.*MAX_N\|MAX_.*=.*max\(.*INITIAL' "$SCRIPT"; then
  echo "LINT WARN: Possible VRAM cap inversion — max(INITIAL, CAP) overrides the memory limit. Use min() to clamp."
fi
if ! grep -q 'CUPY_CACHE_IN_MEMORY' "$SCRIPT"; then
  echo "LINT FAIL: Missing CUPY_CACHE_IN_MEMORY=1. CuPy kernel cache will exhaust BOINC disk limits."
  echo "  Add: os.environ.setdefault('CUPY_CACHE_IN_MEMORY', '1') before cupy import."
  FAIL=1
fi
if [ $FAIL -eq 1 ]; then
  echo "Fix the above before proceeding."
fi
```

**If lint fails:** Fix the flagged lines (replace with correct CuPy-compatible calls), re-run lint. Repeat until clean.

## Step 4: Dry-Run Test (via Local GPU)

Scripts are tested on the project owner's local GPU (RTX 3080 16GB) via a reverse SSH tunnel service on port 9090. This runs the script with an ACTUAL GPU — not CPU fallback — catching cupy-specific bugs, VRAM issues, and kernel launch failures that CPU testing would miss.

For each script that passed syntax and lint:

```bash
SCRIPT="your_experiment.py"
SCRIPT_CONTENT=$(cat /opt/axiom_boinc/html/user/experiments/${SCRIPT})

# Send to local GPU test service via reverse tunnel
RESPONSE=$(curl -s --connect-timeout 10 --max-time 200 -X POST http://localhost:9090/test \
  -H "Content-Type: application/json" \
  -d "$(python3 -c "
import json, sys
print(json.dumps({
    'script_name': '${SCRIPT}',
    'script_content': open('/opt/axiom_boinc/html/user/experiments/${SCRIPT}').read(),
    'run_duration': 1800
}))
")")

echo "GPU test result for ${SCRIPT}:"
echo "$RESPONSE" | python3 -m json.tool
```

**If the GPU service is unreachable** (tunnel down, local PC off), fall back to CPU testing on the server:
```bash
# Fallback: CPU-mode dry-run on server
TESTDIR=$(mktemp -d)
cp /opt/axiom_boinc/html/user/experiments/${SCRIPT} ${TESTDIR}/
echo '{"mode":"experiment","script_url":"test","run_duration":1800,"experiment_name":"exp_test_dryrun"}' > ${TESTDIR}/wu.json
cd ${TESTDIR}
timeout 138 python3 ${SCRIPT}
EXIT_CODE=$?
cd /
rm -rf ${TESTDIR}
```

**Check if GPU service is available first:**
```bash
GPU_SERVICE=$(curl -s --connect-timeout 5 http://localhost:9090/ping 2>/dev/null)
if echo "$GPU_SERVICE" | grep -q '"ok"'; then
  echo "GPU test service available: $(echo $GPU_SERVICE | python3 -c 'import sys,json; print(json.load(sys.stdin).get("gpu","unknown"))')"
  # Use GPU testing (preferred)
else
  echo "GPU test service unavailable — using CPU fallback"
  # Use CPU fallback testing
fi
```

**Interpreting GPU test results (JSON response):**
- `"result": "PASS"` (exit_code 124) = **GOOD** — script ran for the full timeout on the real GPU. Proceed to deploy.
- `"result": "FAST_FINISH"` (exit_code 0) = **FAST FINISH BUG** — script completed too quickly even on GPU. Fix it:
  - Increase initial problem size (GPU scripts should start at N>=512)
  - Ensure GPU FLOPS calibration scales sizes properly
  - Re-test after fixing. Repeat until result is PASS.
- `"result": "SYNTAX_ERROR"` = Fix the syntax error shown in stderr, re-test.
- `"result": "FAST_FINISH"` with non-zero exit_code = **CRASH** — script has a runtime error. Read stderr from the response, identify the bug, fix it, re-test. Common GPU script issues:
  - `NameError: name 'HAS_GPU' is not defined` — script missing the HAS_GPU detection block
  - `cupy.cuda.runtime.CUDARuntimeError` — VRAM issue, reduce problem size or add memory cleanup
  - `rng.random()` instead of `rng.random_sample()`
  - `xp.linalg.eigvals()` — doesn't exist in cupy, use numpy fallback

**SCRIPT FIX RULES (when fixing broken scripts):**
- ONE seed per WU — do NOT add multi-trial loops
- Iterative deepening pattern — fix depth/scaling, not by adding trials
- **Time-aware loops, not fixed counts.** For bisection: `while time < deadline: N *= 2; bisect(N)`.
  For training experiments: `while time < deadline: epoch += 1; train()`. NEVER use fixed `n_epochs` or `N_EPOCHS`.
  This is DIFFERENT from multi-trial loops (which run independent trials and append to a list).
- Memory caps: peak RAM under 4GB, VRAM under 80% of GPU_MEMORY_MB
- Keep existing boilerplate (load_context, build_ordered_experiment_output, NumpyEncoder)
- **`_fitness` must be the FIRST key in `experiment_result`.** If missing, add it. For bisection: `_fitness = 1.0 / max(bracket_width, 1e-15)`. For other types: a meaningful quality scalar (higher = better).
- ALL heavy computation must use cupy (`xp` variable) — do NOT convert to numpy-only
- GPU FLOPS calibration must be present and must scale problem sizes
- `HAS_GPU` detection block must be present
- `os.environ.setdefault("CUPY_CACHE_IN_MEMORY", "1")` must appear before any `import cupy`
- `INITIAL_N = min(INITIAL_N, MAX_N)` — FLOPS-scaled initial size must be clamped to VRAM cap
- Free GPU memory between passes: `cp.get_default_memory_pool().free_all_blocks()`
- **VRAM divisor for dense eigensolve: EIGH_VRAM_DIVISOR = 32768.0.** Divisors below 200 are BANNED -- they cause cudaErrorIllegalAddress on volunteer GPUs. Formula: `MAX_N = int(sqrt(0.80 * GPU_MEMORY_MB * 1e6 / 32768.0))`. If a script uses a smaller divisor (24, 40, 64, 1024, 4096), replace it with 32768.
- **No early bracket convergence exit.** If you see `if bracket_width < 1e-6: break` or similar, remove it. Scripts must keep deepening (doubling N) when the bracket converges, not exit early.

**Also verify the script actually uses cupy:**
```bash
grep -l 'HAS_GPU\|cupy' /opt/axiom_boinc/html/user/experiments/${SCRIPT}
```
If the script does NOT contain HAS_GPU or cupy, it should NOT be deployed as a GPU workunit. Either add GPU support or skip it (let CPU Research handle it instead).

After fixing, always re-run the full test sequence (py_compile → lint → dry-run) before deploying.

## Step 5: Update Work Generator Config

The work generator (`axiom_wu_feeder.py`) creates workunits on-demand using weighted random selection from a config file. It runs every minute via cron, checks the BOINC unsent pool, and generates WUs to fill any deficit. You control which GPU experiments get WUs and in what proportion by editing the config.

**Config file:** `/opt/axiom/experiment_session/wu_generator_config.json`

```json
{
  "gpu_percent": 5,
  "cpu": {"script_name": weight, ...},
  "gpu": {"script_name": weight, ...}
}
```

The `gpu_percent` controls the first roll (chance of generating a GPU vs CPU WU). Within GPU, weights are **relative** — the generator auto-normalizes to 100%. Higher weight = more GPU WUs for that experiment. Adding or removing experiments auto-rescales the rest.

### 5-1. Add Passing Scripts to Config

For each GPU script that passed dry-run testing, add or update it in the GPU section of the config:

```bash
flock -w 30 /tmp/wu_config.lock python3 -c "
import json
config = json.load(open('/opt/axiom/experiment_session/wu_generator_config.json'))
config['gpu']['SCRIPT_NAME'] = 1.0
json.dump(config, open('/opt/axiom/experiment_session/wu_generator_config.json', 'w'), indent=2)
print('Added SCRIPT_NAME to GPU work generator with weight 1.0')
"
```

Weight is your choice. 1.0 is just an example (equal share). Use higher weights (e.g., 2.0, 5.0) if an experiment needs more data faster, or lower weights (e.g., 0.5) to deprioritize it. The generator auto-normalizes all weights to percentages at runtime.

**Only add scripts that contain `HAS_GPU` or `cupy`.** Do NOT add numpy-only scripts to the GPU config.

### 5-2. Verify Config

```bash
python3 -c "
import json
c = json.load(open('/opt/axiom/experiment_session/wu_generator_config.json'))

    print('  ... and %d more' % (len(gpu) - 10))
"
```

```sql
SELECT
  REGEXP_REPLACE(w.name, '_[0-9a-f]{4,8}(_[0-9]+)?$', '') as family_prefix,

  SUM(r.server_state = 5 AND r.outcome = 1) as completed,

JOIN workunit w ON r.workunitid = w.id
WHERE w.name LIKE 'exp_%' AND w.appid = 2

ORDER BY completed DESC;
```

- **Config key naming**: When adding experiments to `wu_generator_config.json`, use the script filename WITHOUT the `.py` extension. e.g. `my_experiment` not `my_experiment.py`. The feeder appends `.py` automatically -- including it causes double-extension 404 errors.

- **cuSOLVER crash protection**: If a GPU script calls `xp.linalg.eigh()`, `eigvalsh()`, `svd()`, or any dense decomposition WITHOUT a try/except around it, add one. On catch: halve the matrix size, cap MAX_N, free GPU memory pool, and continue. cuSOLVER crashes are not OOM errors -- they are illegal memory access kills that abort the entire task. The fix is a 5-line try/except, not a VRAM cap.

If you see a divisor of 24, 40, 64, 1024, or 4096, replace it with 32768.
- **Division-by-zero protection**: Any division by a counter that accumulates inside a loop (e.g. n_batches, n_samples, n_steps) MUST use `max(counter, 1)` as the divisor. When the deadline is tight or already passed, loops may execute zero iterations, leaving counters at 0. This has caused ZeroDivisionError crashes in production.
