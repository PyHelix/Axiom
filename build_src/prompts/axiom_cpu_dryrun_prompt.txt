## Step 2: Syntax Check (py_compile)

```bash
python3 -m py_compile /opt/axiom_boinc/html/user/experiments/SCRIPT.py
```

## Step 3: Dry-Run Test

For each script that passed syntax check, run a timed dry-run:

```bash
SCRIPT="your_experiment.py"
TESTDIR=$(mktemp -d)
cp /opt/axiom_boinc/html/user/experiments/${SCRIPT} ${TESTDIR}/
echo '{"mode":"experiment","script_url":"test","run_duration":900,"experiment_name":"exp_test_dryrun"}' > ${TESTDIR}/wu.json
cd ${TESTDIR}
timeout 138 python3 ${SCRIPT}
EXIT_CODE=$?
cd /
rm -rf ${TESTDIR}
```

**Interpreting results:**
- **Exit code 124** (timeout killed it) = **GOOD** — script would run for the full 15 minutes. Proceed to deploy.
- **Exit code 0** (finished before timeout) = **FAST FINISH BUG** — script completed in under 2.3 minutes. This wastes volunteer compute. Fix it:
  - Remove early convergence breaks
  - Increase initial problem size (N should start at 64+ for iterative deepening)
  - Add more bisection steps
  - Ensure time estimation doesn't abort too early
  - Re-test after fixing. Repeat until exit code is 124.
- **Any other exit code** = **CRASH** — script has a runtime error. Read stderr, identify the bug, fix it, re-test. Common issues:
  - NameError: undefined variable or missing import
  - TypeError: wrong argument types to numpy functions
  - KeyError: missing key in wu.json context
  - AttributeError: using scipy/torch functions (CPU scripts must be numpy-only)

**SCRIPT FIX RULES (when fixing broken scripts):**
- ONE seed per WU — do NOT add multi-trial loops
- Iterative deepening pattern — fix depth/scaling, not by adding trials
- **Time-aware loops, not fixed counts.** For bisection: `while time < deadline: N *= 2; bisect(N)`.
  For training experiments: `while time < deadline: epoch += 1; train()`. NEVER use fixed `n_epochs` or `N_EPOCHS`.
  This is DIFFERENT from multi-trial loops (which run independent trials and append to a list).
- Memory caps: peak RAM must stay under 2GB
- Keep existing boilerplate (load_context, build_ordered_experiment_output, NumpyEncoder)
- **`_fitness` must be the FIRST key in `experiment_result`.** If missing, add it. For bisection: `_fitness = 1.0 / max(bracket_width, 1e-15)`. For other types: a meaningful quality scalar (higher = better).
- If script runs a subset of a config grid per WU, verify summary code guards missing configs with `.get()` or existence checks
- numpy + stdlib only — no scipy, torch, matplotlib
- Early submit is OK only if bracket converges to machine precision, NOT because problem is too small
- **No early bracket convergence exit.** If you see `if bracket_width < 1e-6: break` or similar, remove it. Scripts must keep deepening (doubling N) when the bracket converges, not exit early.

After fixing, always re-run the full test sequence (py_compile → dry-run) before deploying.

## Step 4: Update Work Generator Config

The work generator (`axiom_wu_feeder.py`) creates workunits on-demand using weighted random selection from a config file. It runs every minute via cron, checks the BOINC unsent pool, and generates WUs to fill any deficit. You control which experiments get WUs and in what proportion by editing the config.

**Config file:** `/opt/axiom/experiment_session/wu_generator_config.json`

```json
{
  "gpu_percent": 5,
  "cpu": {"script_name": weight, ...},
  "gpu": {"script_name": weight, ...}
}
```

Weights are **relative** — the generator auto-normalizes to 100%. Higher weight = more WUs generated for that experiment. For example, if you have 3 experiments with weights 1.0, 2.0, 3.0, they get ~17%, ~33%, ~50% of new WUs respectively. Adding or removing experiments auto-rescales the rest.

### 4-1. Add Passing Scripts to Config

For each script that passed dry-run testing, add or update it in the CPU section of the config:

```bash
flock -w 30 /tmp/wu_config.lock python3 -c "
import json
config = json.load(open('/opt/axiom/experiment_session/wu_generator_config.json'))
config['cpu']['SCRIPT_NAME'] = 1.0
json.dump(config, open('/opt/axiom/experiment_session/wu_generator_config.json', 'w'), indent=2)
print('Added SCRIPT_NAME to CPU work generator with weight 1.0')
"
```

Weight is your choice. 1.0 is just an example (equal share). Use higher weights (e.g., 2.0, 5.0) if an experiment needs more data faster, or lower weights (e.g., 0.5) to deprioritize it. The generator auto-normalizes all weights to percentages at runtime.

### 4-2. Verify Config

```bash
python3 -c "
import json
c = json.load(open('/opt/axiom/experiment_session/wu_generator_config.json'))
cpu = {k: v for k, v in c['cpu'].items() if v > 0}
total = sum(cpu.values())
print('Active CPU experiments: %d' % len(cpu))
for name, weight in sorted(cpu.items(), key=lambda x: -x[1])[:10]:
    print('  %s: %.1f (%.1f%%)' % (name, weight, 100*weight/total))
if len(cpu) > 10:
    print('  ... and %d more' % (len(cpu) - 10))
"
```

**Do NOT stage JSON files in `/opt/axiom_boinc/stage/cpu/`.** The work generator handles all WU creation automatically. Do NOT call `bin/create_work` directly.

## Step 5: Retire Over-Seeded Experiments (CPU only)

**Check for experiments that should be retired.**

```sql
SELECT
  REGEXP_REPLACE(w.name, '_[0-9a-f]{4,8}(_[0-9]+)?$', '') as family_prefix,
  COUNT(DISTINCT w.id) as total_wus,
  SUM(r.server_state = 5 AND r.outcome = 1) as completed,
  SUM(r.server_state = 2) as unsent,
  SUM(r.server_state = 4) as running
FROM result r
JOIN workunit w ON r.workunitid = w.id
WHERE w.name LIKE 'exp_%' AND w.appid = 1
GROUP BY family_prefix
HAVING unsent > 0 OR running > 0
ORDER BY completed DESC;
```

**Retirement is YOUR judgment call.** Consider:
- Has the finding already been confirmed in findings_summary.txt?
- Is the statistical evidence strong enough for a conclusion?
- Is this a well-known/textbook result?
- Is the experiment failing across many hosts (not just one)?
- Have results converged?

If you decide an experiment is done, **record it in the retirement registry.** Do NOT abort WUs yourself — The cleanup cron handles all actual aborting (runs every 5 minutes).

```bash

```

Also set the experiment's weight to 0 in the work generator config so no new WUs are created:
