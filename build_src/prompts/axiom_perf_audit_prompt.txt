# Axiom BOINC — Error Triage 1 — Slow Tasks (Step 11 of 14)

All experiment tasks target 15 minutes. Over 50 minutes means the script has a performance bug.

## Step 1: Find Slow Experiments

```sql
SELECT
  SUBSTRING_INDEX(SUBSTRING_INDEX(r.name, '_gpu_', 1), '_cpu_', 1) as experiment,
  COUNT(*) as slow_tasks,
  ROUND(AVG(r.elapsed_time)) as avg_sec,
  ROUND(MAX(r.elapsed_time)) as max_sec
FROM result r
WHERE r.server_state=5 AND r.outcome=1
  AND r.elapsed_time > 3000
  AND r.received_time > UNIX_TIMESTAMP() - 36000
GROUP BY experiment
ORDER BY max_sec DESC;
```

If no results: log "No slow experiments" and exit.

## Step 2: Fix the Script

For each slow experiment:

1. Find the script: `ls /opt/axiom_boinc/html/user/experiments/*EXPERIMENT_NAME*.py`
2. Read it and figure out why it's slow
3. Fix it so it completes within 15 minutes
4. Keep changes minimal — only fix the performance issue

### SCRIPT FIX RULES — you MUST follow these when editing any experiment script:

- **ONE SEED PER WORKUNIT.** Each WU runs ONE deep computation with a single seed.
  Do NOT add multi-trial loops (independent trials: run_trial(); all_results.append()).
  NOTE: `while time.time() < deadline: train_one_epoch()` is CORRECT for training experiments.
  The BAD pattern runs INDEPENDENT trials. The GOOD pattern is one continuous computation that deepens.
  Also do NOT use fixed epoch counts (`n_epochs = 120`) — use time-aware loops.
  Aggregation happens at the BOINC level across hundreds of WUs with different seeds.
- **ITERATIVE DEEPENING, NOT MULTI-TRIAL.** If a script is slow because the problem
  size is too large, fix by using iterative deepening: start small, double the problem
  size. For training experiments, fix by using `while time.time() < deadline:` instead of fixed epochs.
  size each pass, estimate if next pass fits in the time budget (last_pass_time * 8),
  stop if it won't. Write a checkpoint after each completed pass.
- **MEMORY CAP: 2GB for CPU (appid=1), 4GB for GPU (appid=2).** If the script is slow
  because it allocates huge arrays, shrink arrays and add more iterations instead.
  "Maximize depth" means more iterations over fixed memory, NOT bigger arrays.
- **EARLY SUBMIT.** If the computation stabilizes (output stops changing) or hits memory
  cap before 15 minutes — stop and submit. Do NOT pad with busy-wait or dummy loops.
- **Keep the `build_ordered_experiment_output()` boilerplate and `RESULT_SCHEMA`.**
  Do NOT remove or modify the output serialization machinery.
- **Seeds are injected automatically.** The WU feeder adds a unique random seed to each
  workunit JSON. The client injects it as `EXPERIMENT_SEED` into the script namespace.
  The `load_context()` function picks it up. Do NOT change the seed handling.
