STEP 1 — ERROR RATE PER EXPERIMENT

Run this query to see error rates by experiment family:
  SELECT
    REGEXP_REPLACE(w.name, '_[0-9a-f]{4,8}(_[0-9]+)?$', '') as family,
    w.appid,
    SUM(r.server_state=5 AND r.outcome=1) as ok,
    SUM(r.server_state=5 AND r.outcome<>1) as err,
    SUM(r.server_state=2) as unsent,
    SUM(r.server_state=4) as in_progress,
    ROUND(100.0 * SUM(r.server_state=5 AND r.outcome<>1) /
      NULLIF(SUM(r.server_state=5), 0), 1) as err_pct
  FROM result r
  JOIN workunit w ON r.workunitid=w.id
  WHERE w.name LIKE 'exp_%'
    AND r.received_time > UNIX_TIMESTAMP() - 36000
  GROUP BY family, w.appid
  HAVING (err > 5 AND err_pct > 5) OR err > 50
  ORDER BY err_pct DESC;


STEP 1A — CLIENT ERROR TELEMETRY

v6.39+ clients report script errors directly to the server before BOINC even sees them.
This catches errors that BOINC might mark as "success" (exit code 0) because the client
handles the exception gracefully. The telemetry log is the FASTEST way to spot broken scripts.

Log file: /opt/axiom/experiment_results/error_telemetry.jsonl
Format: one JSON object per line with fields: ts, experiment_name, error_type, error_message, platform, client_version, has_gpu, gpu_name, gpu_memory_mb, ip

Summarize errors from the last 12 hours:
  python3 -c "
import json, time
from collections import Counter
cutoff = time.time() - 43200
by_script = Counter()
by_type = Counter()
details = {}
try:
    for line in open('/opt/axiom/experiment_results/error_telemetry.jsonl'):
        d = json.loads(line)
        from datetime import datetime
        ts = datetime.strptime(d['ts'], '%Y-%m-%dT%H:%M:%SZ').timestamp()
        if ts < cutoff:
            continue
        # Strip hex suffix to get script name
        parts = d['experiment_name'].rsplit('_', 1)
        script = parts[0] if len(parts) == 2 and len(parts[1]) == 6 else d['experiment_name']
        key = (script, d['error_type'])
        by_script[script] += 1
        by_type[d['error_type']] += 1
        if key not in details:
            details[key] = d['error_message'][:200]
print('=== Error Telemetry (last 12h) ===')
print(f'Total reports: {sum(by_script.values())}')
print()
print('By script (top 15):')
for name, count in by_script.most_common(15):
    print(f'  {count:4d}  {name}')
print()
print('By error type:')
for etype, count in by_type.most_common():
    print(f'  {count:4d}  {etype}')
print()
print('Unique script+error combinations:')
for (script, etype), msg in sorted(details.items(), key=lambda x: -by_script[x[0][0]]):
    print(f'  {script}')
    print(f'    {etype}: {msg}')
except FileNotFoundError:
    print('No telemetry log found yet (no v6.39+ errors reported)')
"

TRIAGE RULES FOR TELEMETRY ERRORS:
- AttributeError / NameError / TypeError / ImportError → almost always a FIXABLE script bug. Read the script source and fix it.
- CUDARuntimeError (cudaErrorIllegalAddress, cudaErrorLaunchFailure) → usually VRAM overflow. Check the script's MAX_N / VRAM sizing formula. If the divisor is too small (< 200), increase it.
- OutOfMemoryError → same as above, VRAM sizing too aggressive.
- CUDADriverError → hardware/driver issue on the volunteer's machine. SKIP unless it affects many IPs.
- URLError / ConnectionError → network issue. SKIP.
- Add telemetry-discovered broken scripts to your Step 2 triage list alongside BOINC DB errors.
- IMPORTANT: CuPy's RandomState does NOT have .random() — use .random_sample() instead. CuPy does NOT have linalg.eig() or linalg.eigvals() — use numpy fallback.


STEP 1B — SILENT FAILURE DETECTION

Some experiments crash gracefully — the client catches the error, writes a minimal
result (just metadata + elapsed time), and exits with code 0. BOINC marks these as
outcome=1 (success). They waste compute but never appear in Step 1's error query.

Detect them by checking sanitized result files for experiments with recent validated results:

a) Get active experiment families with recent validated results:
  SELECT
    REGEXP_REPLACE(w.name, '_[0-9a-f]{4,8}(_[0-9]+)?$', '') as family,
    COUNT(*) as validated_24h
  FROM result r
  JOIN workunit w ON r.workunitid=w.id
  WHERE r.outcome=1 AND r.validate_state=1
    AND r.received_time > UNIX_TIMESTAMP() - 36000
    AND w.name LIKE 'exp_%'
  GROUP BY family
  HAVING validated_24h > 20
  ORDER BY validated_24h DESC;

b) For each family with >20 validated results in 24h, sample 3 recent sanitized files
   and check if they contain actual experiment data or just metadata + elapsed:

  python3 -c "

  family = sys.argv[1]
  files = sorted(glob.glob(f'/opt/axiom/experiment_results/sanitized/{family}_*_safe.json'),
                 key=os.path.getmtime, reverse=True)[:3]

      print(f'{family}: no sanitized files found')
      sys.exit(0)

      if not has_data:

      print(f'  {os.path.basename(f)}: data_keys={data_keys}, has_science={has_data}')

      print(f'>>> SILENT FAILURE: {family} -- all sampled results are empty')
  " FAMILY_NAME

STEP 1C — FAST FINISH DETECTION

Experiments should run for ~900 seconds (15 minutes). Scripts that finish in under
2 minutes are broken — they are not using the full time budget to maximize compute depth.
This wastes scheduler overhead and produces shallow, low-resolution results.

Detect them (only active experiments that still have unsent or in-progress tasks):
  SELECT
    f.family, f.appid, f.n, f.avg_elapsed, f.min_elapsed, f.max_elapsed
  FROM (
    SELECT
      REGEXP_REPLACE(w.name, '_[0-9a-f]{4,8}(_[0-9]+)?$', '') as family,
      w.appid,
      COUNT(*) as n,
      ROUND(AVG(r.elapsed_time), 1) as avg_elapsed,
      ROUND(MIN(r.elapsed_time), 1) as min_elapsed,
      ROUND(MAX(r.elapsed_time), 1) as max_elapsed
    FROM result r
    JOIN workunit w ON r.workunitid=w.id
    WHERE r.outcome=1 AND r.validate_state=1
      AND r.received_time > UNIX_TIMESTAMP() - 36000
      AND w.name LIKE 'exp_%'
    GROUP BY family, w.appid
    HAVING avg_elapsed < 120 AND n >= 5
  ) f
  WHERE EXISTS (
    SELECT 1 FROM result r2
    JOIN workunit w2 ON r2.workunitid=w2.id
    WHERE REGEXP_REPLACE(w2.name, '_[0-9a-f]{4,8}(_[0-9]+)?$', '') = f.family
      AND r2.server_state IN (2, 4)
  )
  ORDER BY f.avg_elapsed ASC;

For each fast-finishing family:
a) Read the script from /opt/axiom_boinc/html/user/experiments/
b) The script should use the full run_duration to maximize compute depth (larger grids,
   more iterations, finer resolution) for ONE seed. Each workunit = one deep computation.
   If it runs a tiny fixed problem and exits early, that is the bug.
c) FIX the script: scale the problem size based on run_duration so the CPU/GPU stays
   pinned at ~100% utilization for the full 15 minutes. Do NOT add a multi-trial loop â
   instead increase grid size, iteration count, or sample size.
d) Add fast-finish families to the Step 2 triage list with action FIX.

STEP 1D — GPU WRAPPER DETECTION (appid=2 only)

GPU experiments must do their heavy computation in cupy, not numpy. Some scripts
cheat by importing numpy for 95% of the work and only calling cupy for one
operation (e.g., eigenvalue decomposition). These run at 0% GPU utilization
while occupying a GPU slot.

Detect them:
a) List all active GPU experiment scripts (appid=2 with unsent or in-progress tasks)
b) For each script, read the source from /opt/axiom_boinc/html/user/experiments/
c) Check: does the script import numpy as np and use np.* for the main computation
   loop while only using cp/cupy for 1-2 calls? Signs of a wrapper:
   - 'import numpy as np' at top level with np.* used throughout the computation
   - cupy/cp used only for linalg, eigenvalues, or array creation
   - Results immediately converted back via cp.asnumpy() after one GPU call
d) If a script is a cupy wrapper: ABORT all its unsent and in-progress tasks.
   The experiment concept may be valid but needs to be redesigned with cupy
   throughout by the GPU research step. Do NOT rewrite it yourself.
