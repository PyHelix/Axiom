NEW EXPERIMENT: grokking_dynamics_v6.py — DESIGN RATIONALE
============================================================

MOTIVATION:
  v5 FAILED to grok despite fixing the seeding bug. The root cause is
  now clear: weight_decay=3.0 was TOO STRONG.

  WEIGHT NORM COMPARISON ACROSS ALL VERSIONS:
    v1: P=97, wd=0.0  → wn=high, NEVER memorized (4.87% train)
    v2: P=97, wd=1.0  → wn=157.5, decline=12%, test=49% at 100K ✓
    v3: P=53, wd=1.0, lr=0.003 → wn=97.4, decline=7%, test=3%
    v4: P=23, wd=1.0, hidden=64 → wn=51.9, decline=4%, test=0%
    v5: P=23, wd=3.0, hidden=128 → wn=31.2, decline=0.1%, test=0%

  THE PATTERN IS CLEAR:
    Weight norm peak correlates with grokking success.
    v2 with wn=157.5 showed grokking; all others with wn<100 failed.
    The critical weight norm threshold appears to be ~100-150.

  v5's failure confirms that STRONGER weight decay does NOT help.
  It actually PREVENTS grokking by limiting the model's ability to
  build rich representations during memorization.

v6 DESIGN:
  Same as v2 (which showed actual grokking progress) but with P=23:
  - P=23 (smallest useful prime, fast dynamics)
  - hidden_dim=128 (same as v2, proven capacity)
  - weight_decay=1.0 (BACK TO v2's value — the only version that worked)
  - lr=0.001 (same as v2)
  - GELU activation (keep from v5, not the bottleneck)
  - 500K epochs (5x more than v2's 100K)
  - v5's seeding fix (confirmed working)

PREDICTION:
  With P=23 and wd=1.0, hidden=128: weight norm should peak at ~100-150
  (similar to v2's 157). Since P=23 has only 529 examples (vs P=97's
  9409), the model should grok FASTER — potentially within 50K-200K
  epochs rather than v2's incomplete 100K+ trajectory.

  If weight norm peaks above 100 and shows >10% decline, grokking is
  likely. If test accuracy reaches 50%+, it's in progress. If 95%+,
  GROKKING IS CONFIRMED.

  v6 is the MINIMAL change from v5: only weight_decay 3.0 → 1.0.
  This directly tests the hypothesis that wd=1.0 is the critical value.
